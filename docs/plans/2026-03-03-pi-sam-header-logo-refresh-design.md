# pi-sam 헤더 로고 개선 디자인

## 배경
현재 `pi-sam` 시작 헤더는 상단 3줄 심볼(◤◥/◢██◣/◥██◤) 기반이라 브랜드 텍스트 중심 인지가 약하다. 사용자는 헤더를 더 그럴싸하게 보이도록, `pi-sam` 자체를 ASCII/유니코드 블록 문자로 표현한 텍스트 로고 형태를 요청했다.

## 요구사항 확정
- 로고 형태: 텍스트 로고형
- 로고 텍스트: `pi-sam` (소문자)
- 스타일: 블록형 (`█`, `▀` 계열)
- 크기: Compact (대략 폭 20~28자, 높이 3~4줄)
- 기존 메타 라인(`/help · /model · /resume  v<version>`)은 유지

## 접근안 비교
1. 하드코딩 로고 교체
   - 장점: 변경 범위 최소, 안정적, 즉시 반영 가능
   - 단점: 스타일 확장 시 코드 수정 필요
2. 로고 프리셋(테마) 구조 도입
   - 장점: 향후 확장 용이
   - 단점: 현재 요구 대비 과설계 가능성
3. 동적 생성기 방식
   - 장점: 자동화/확장성 높음
   - 단점: 복잡도 증가, 렌더링 일관성 리스크

## 선택
- **선택안: 1번(하드코딩 로고 교체)**
- 이유: 단일 스타일 요구가 명확하고, 기존 테스트/렌더링 흐름을 거의 건드리지 않으면서 목표 달성이 가능하다.

## 설계
### Architecture
- 대상: `src/extensions/branding-header.ts`
- `getPiSamHeaderLines(version)`의 상단 로고 라인을 Compact 블록형 `pi-sam`으로 교체
- 색상 처리(`index <= 2`, `index === 3`)는 기존 로직을 최대한 유지

### Components
- 로고 라인 3~4줄로 구성
- 브랜드 텍스트(`pi-sam`)와 메타 라인 유지
- 총 라인 수는 테스트 규칙(5~7줄) 준수

### Data Flow
- `session_start` → `setHeader()` → `getPiSamHeaderLines(VERSION)` → 테마 색상 적용 후 렌더
- 데이터 흐름 변화 없음, 표시 문자열만 변경

### Error/Compatibility
- Compact 폭 유지로 좁은 터미널에서도 줄바꿈 리스크 최소화
- 블록 문자 폰트 차이 가능성을 고려해 지나치게 복잡한 패턴은 회피

### Testing
- `src/extensions/branding-header.test.ts` 기준 회귀 확인
  - 줄 수 5~7
  - `pi-sam` 포함
  - `/help`, `/model`, `/resume`, 버전 포함
- 필요 시 로고 라인 존재성 정도의 최소 검증만 추가
